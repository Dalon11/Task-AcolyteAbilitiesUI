# Task-AcolyteAbilitiesUI

## Описание
Тестовый Unity-проект интерфейса окна управления персонажем: `Abilities` и `Modifications`.

Цель: реализовать UI, который выглядит как финальная игровая версия (а не прототип), и демонстрирует работу выбора персонажа, совместимость модификаторов и их назначение на способности.

## Основной функционал
1. Переключение персонажей в блоке `Party`.
2. При смене персонажа обновляются:
- портрет
- имя
- HP и Armor
- список Abilities
- список Modifications
- Action Bar
3. Для каждого персонажа хранится собственный набор Abilities и Modifications.
4. Наборы не пересоздаются при повторном выборе персонажа.
5. У разных персонажей разная заполненность списков:
- от пустого/почти пустого
- до максимально заполненного

## Логика Abilities и Modifications
1. Один модификатор можно назначить только одной способности одновременно.
2. Если модификатор уже назначен, его нельзя назначить второй способности, пока он не снят.
3. Разрешён перенос модификатора с одной способности на другую.
4. Не все модификаторы совместимы со всеми способностями.
5. Типы модификаторов:
- `Psyker`
- `Dot`
- `Attack`
- `Buff`

## Подсветка совместимости
1. При наведении на Ability подсвечиваются совместимые Modifications.
2. При наведении на Modification подсвечиваются совместимые Abilities.
3. Визуальные состояния должны явно показывать:
- какие модификаторы уже заняты
- какие доступны для назначения

## Взаимодействие
1. Обязательный способ назначения: `Drag & Drop`.
2. Дополнительные способы допустимы, если улучшают UX.

## UX требования
1. Интерфейс должен быть понятен без обучения.
2. Игрок должен сразу понимать:
- как назначать модификаторы
- какие модификаторы заняты
- какие способности совместимы
3. Допускается использование:
- подсветки
- разных визуальных состояний
- tooltip
- коротких текстовых подсказок

## Технические требования
1. Целевое разрешение: `1920x1080`.
2. Полная адаптивность не обязательна.
3. При изменении разрешения UI не должен ломаться:
- элементы не накладываются
- элементы не выходят за экран
4. Верхняя навигация (Inventory, Factions, Character и т.д.) может быть статичной заглушкой.

## Архитектура
Реализация должна соответствовать `MVVM`.

Дополнительные правила проекта:
1. Следовать `SOLID`.
2. Не размещать бизнес-логику во `View`.
3. `MonoBehaviour` использовать только для View-слоя.
4. Логику размещать в обычных классах с явными зависимостями (через конструктор).

## Ограничения репозитория
1. Код изменяется только в `Assets/Feature` и дочерних папках.
2. Конфиги и данные размещаются в `Assets/Game/SO`.
3. Сцены размещаются в `Assets/Scenes`.

## Критерии готовности
1. Переключение персонажей работает корректно.
2. Для каждого персонажа сохраняется свой стабильный набор Abilities/Modifications.
3. Назначение модификаторов учитывает занятость и совместимость.
4. Подсветка совместимости работает в обе стороны (Ability -> Modifications и наоборот).
5. Интерфейс визуально читаем и не распадается при изменении разрешения.
6. Архитектура соответствует MVVM и ограничениям проекта.

## Материалы
Графические ассеты и макеты интерфейса предоставляются отдельно вместе с заданием.
